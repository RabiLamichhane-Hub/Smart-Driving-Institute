Smart Driving Institute Management System — a web-based platform that digitizes and automates the operations of a driving training institute. It replaces manual paperwork and scattered records with a centralized system accessible by admins, instructors, and trainees. Technology Stack Built with Django (Python) on the backend, PostgreSQL as the database, and plain HTML/CSS with Bootstrap for the frontend. JavaScript and AJAX are used for dynamic interactions like the vehicle dropdown filtering. The project follows Django's MTV (Model-Template-View) pattern. Project Structure The project is organized as a multi-app Django project under the namespace SDIMS_apps. Each module is a separate Django app: AppPurposeaccountsCustom user model, login, logout, role managementtraineesTrainee registration, profiles, statusinstructorsInstructor registration, profilescoursesCourse creation, vehicle type filteringvehiclesVehicle inventory managementtrainingLesson scheduling (in progress)license_mocktestMock theory test with question bankaccountingFee records, payment trackinghomesandallHome page and general views User Roles Three distinct actors with different permissions: Admin — full control over everything. Creates instructors and trainees, manages courses and vehicles, records payments, views all test results. Only admin can delete records. Instructor — limited operational access. Views and edits their assigned trainees, marks lessons as completed, records payments when collecting cash, views test results of their trainees. Trainee — read-mostly access. Views their own profile, enrolled course, lesson schedule, payment status, and takes mock tests. Core Modules Accounts Custom User model extending Django's AbstractUser with added fields — role, phone, address, must_change_password. Handles login, logout, role-based redirects, and a forced password change on first login. Username and password are auto-generated when an admin creates a new user. Trainees Trainee model linked OneToOneField to User. Stores trainee-specific data — gender, date of birth, enrolled course, enrollment date, status, guardian info. Status flows through ENROLLED → TRAINING → COMPLETED / FAILED. Instructors Instructor model linked OneToOneField to User. Stores license number, experience, availability, and status. Courses Course model with vehicle type (Car, Bike, Scooter), level (Beginner, Intermediate, Advanced), duration, total lessons, fee, and a ForeignKey to a single Vehicle. The course creation form uses AJAX to dynamically filter the vehicle dropdown based on selected vehicle type. Backend validation ensures the selected vehicle matches the course vehicle type. Vehicles Vehicle model with brand, model, registration number, bluebook number, engine number, chassis number, vehicle type, transmission, fuel type, status, insurance expiry, last service date, and dual control flag. Training (Lesson Scheduling) LessonSchedule model linking a trainee, instructor, and vehicle at a specific date and time. Has status choices — scheduled, ongoing, completed, cancelled. Includes conflict checking to prevent double-booking the same instructor or vehicle. Not fully built yet. Mock Test Question model with four options and a correct answer stored as Option_A, Option_B etc. Questions are imported from a CSV file via a custom management command. The test generates 25 random questions per session, stores them in Django's session to prevent regeneration on refresh, and scores the result with a pass mark of 16/25. Results are stored in a TestAttempt model linked to the user for history tracking. Accounting FeeRecord model with OneToOneField to Trainee, automatically pulls the course fee on creation. Tracks total paid, remaining balance, and status (unpaid/partial/paid). Payment model records each installment with amount, method (cash/online/bank transfer), received by, date, and notes. Status updates automatically after each payment. Key Technical Decisions Made Multi-app over single-app — each module is isolated in its own Django app for cleaner code, easier debugging, and independent development. ForeignKey over ManyToManyField for vehicle on course — originally M2M, changed to FK since only one vehicle per course is needed. This simplified forms, validation, and templates significantly. Auto-generated credentials — admins never manually set usernames or passwords. generate_username() uses the phone number and generate_password() generates a random secure password. Credentials are shown once on a user_created.html page. AJAX vehicle filtering — when vehicle type is selected in the course form, a fetch call hits /courses/ajax/vehicles/?vehicle_type=car and dynamically repopulates the vehicle dropdown. The backend validates the match again on save. Session-based mock test — questions are stored in the Django session on first load and reused on refresh. A "New Test" button explicitly clears the session to regenerate questions. A JavaScript beforeunload event warns the user before leaving mid-test. FeeRecord auto-population — overriding save() on FeeRecord to pull total_fee from trainee.course.fee automatically on creation so admins never manually enter the fee.
